  • 4

💡
Arithmetic Intensity = FLOPs / Memory Bytes
이 값이 Ridge Point보다 낮으면 → 메모리 바운드

메모리 병목이란, 연산 유닛(CUDA Core)은 빠른데 데이터를 공급하는 메모리가 느려 연산 유닛이 대부분의 시간을 기다리며 보내는 상태입니다. LLM inference(batch=1)는 각 토큰 생성 시 모델 가중치 전체를 메모리에서 읽어야 하므로, 연산 강도(Arithmetic Intensity)가 매우 낮습니다.

Arithmetic Intensity (산술 강도) = FLOPs ÷ DRAM Accesses (Bytes).
Transformer decode(batch=1): weight matrix W [d×d] → FLOPs = 2d² (gemv), Bytes = 2d² (FP16 load) → AI ≈ 1.
Prefill (batch=B, seq=L): FLOPs = 2BLd², Bytes = 2d² → AI = BL. 배치가 커질수록 compute-bound에 가까워집니다.
따라서 inference 최적화 핵심 = (1) 배치 증가 + (2) 메모리 대역폭 증가.

DRAM Physics · Memory Fundamentals
DRAM — 저장의 물리학
1T1C 셀 구조, Refresh, 그리고 레이턴시 vs 대역폭의 본질적 차이
  • T
    1T1C 셀: 트랜지스터(T) 1개 + 커패시터(C) 1개. 커패시터 충전 = 1, 방전 = 0. 셀 면적 최소화 → 고용량 가능
  • R
    Refresh 필요성: 커패시터는 누설 전류(leakage current)로 서서히 방전. 통상 64ms마다 전체 리프레시 수행 (JEDEC 표준)
  • L
    레이턴시: Row Activation → Column Access 소요 시간. 물리적 charge/discharge 시간에 종속 → 주파수로 단축 한계 있음
⚡
레이턴시 ≠ 대역폭
레이턴시 = 첫 데이터까지 대기 시간 (ns 단위)
대역폭 = 초당 전송량 (GB/s) — 버스 폭 × 주파수로 결정

DRAM은 휘발성(volatile) 메모리로, 전원이 끊기면 데이터가 사라집니다. 커패시터 기반 셀이므로 주기적 Refresh가 필수입니다. Refresh 중에는 해당 Row에 접근이 불가능하여 미세한 성능 저하가 발생합니다. 현대 DDR5는 온칩 ECC를 통해 single-bit 오류를 자동 수정합니다.

CL-tRCD-tRP-tRAS 타이밍 파라미터: CL36-46-46-113 (DDR5-6400)
CL(CAS Latency): Column Address Strobe 활성화 후 첫 데이터까지 클럭 수 (36클럭 × 0.312ns = 11.25ns)
tRCD: RAS→CAS 딜레이 — Row Activation 후 Column 접근 가능까지 대기 (46클럭)
tRP: Row Precharge 시간 — 다음 Row 활성화 전 Precharge 완료까지 대기 (46클럭)
숫자가 작을수록 빠름. DDR5 고속에서 클럭 증가로 절대 ns 값은 DDR4와 유사하거나 약간 증가.

DDR5는 DDR4 대비 단순 클럭 증가가 아닌 버스 아키텍처 자체를 재설계했습니다. 특히 Sub-channel 분리는 독립적 메모리 접근을 가능하게 해 AI 추론 시 가중치 로딩 효율을 높입니다.

DDR4는 64-bit 단일 채널. DDR5는 각 32-bit 채널이 독립적 커맨드/어드레스 버스를 가집니다. 서로 다른 Row를 동시에 활성화 가능 → Bank Conflict 감소. Burst Length도 BL8→BL16으로 증가해 연속 전송 효율 향상.

2-socket Intel Xeon (Sapphire Rapids+): 소켓당 8채널 DDR5 → 총 16채널.
16채널 × 67.2 GB/s (DDR5-8400) = ~1.07 TB/s CPU 메모리 대역폭.
LLM 서빙에서 CPU 메모리는 GPU VRAM 오프로드 대상 — DDR5 대역폭이 클수록 CPU-offload 추론 속도 향상.

LPDDR(Low Power DDR)는 모바일 기기 특화 메모리입니다. DIMM 형태가 아닌 BGA(Ball Grid Array)로 SoC 기판에 직접 납땜합니다. 전압이 1.1V (vs DDR5 1.1V이지만 구동 회로 차이로 실효 소비전력 차이 있음), 배터리 기기에서 전력 효율 최대화.

BGA 패키징: 메모리 칩을 SoC 패키지 위에 직접 적층(PoP: Package on Package) 또는 SoC 옆에 같은 기판에 배치. 신호 경로가 극도로 짧아 임피던스 최소화 → 고속 신호 전송 가능. 단점: 메모리 용량 교체/확장 불가능. Soldered-down 구조.

Apple Unified Memory Architecture (UMA): CPU, GPU, Neural Engine, ISP가 동일 물리 메모리 풀 접근. 기존 PCIe 방식 (CPU DDR ↔ GPU GDDR, 수십 GB/s PCIe BW)과 달리 복사 오버헤드 0. LLM 추론 시 GPU VRAM 제한 없이 시스템 전체 메모리 활용 가능 → M4 Max 128GB 전량을 LLM에 사용 가능.

GDDR(Graphics DDR)는 GPU 전용 메모리입니다. DRAM 칩을 GPU PCB에 직접 솔더링하며, HBM과 달리 2D 배치로 면적 효율은 낮지만 비용 효율이 높고 용량 확장이 용이합니다.

PAM4 (Pulse Amplitude Modulation 4-level): 기존 NRZ(Non-Return-to-Zero, 0/1 2레벨) 대신 0/1/2/3 4레벨 신호를 사용. 같은 물리적 주파수에서 2배 데이터 전송. 단, SNR(신호 대 잡음비) 요구가 높아 정밀한 아날로그 회로 설계 필요. SerDes 복잡도 증가.

GDDR 선택: 저비용, PCB 면적 대비 용량 확장 용이, 양산 용이. 소비자 GPU, 추론 가속기(비HBM)에 적합.
HBM 선택: 고대역폭 (HBM3e 4.8TB/s vs GDDR7 1.8TB/s), 소형 패키지, 단 CoWoS 특수 패키징 필수 → 비용 10× 이상, TSMC CoWoS 생산 능력이 병목.

HBM(High Bandwidth Memory)은 DRAM 다이를 수직으로 적층하고 TSV(Through-Silicon Via)로 연결합니다. 핀당 대역폭보다 총 핀 수(버스 폭)를 극대화하는 전략. HBM3e는 핀당 6.4 Gbps × 1,024bit = 819.2 GB/s per stack (JEDEC 기준, 제조사별 차이 있음).

TSV(Through-Silicon Via): 실리콘 웨이퍼를 관통하는 수직 전도 경로. 지름 ~5μm, 수만 개 TSV를 통해 1,024~2,048bit 와이드 버스 구현. 마이크로 범프로 다이 간 본딩 → 적층 수 12~16단까지 가능. 열 방출이 핵심 과제.

TSMC CoWoS(Chip on Wafer on Substrate): GPU 다이와 HBM 스택을 실리콘 인터포저 위에 함께 배치. 인터포저가 고밀도 배선 제공 (GPU↔HBM 신호 거리 최소화). CoWoS-L(Large)는 레티클 2~4배 크기 인터포저 → CoWoS 생산 능력이 AI 칩 공급 병목의 핵심 원인.

HBM 공급업체는 전 세계 3곳뿐: SK하이닉스(H200/H100 주요 공급, HBM3e 점유율 1위), Samsung(HBM3e QC 이슈로 후발), Micron(HBM3e 2024년 진입). 수요 폭증 대비 공급 부족으로 2024~2025년 AI 칩 공급 제약의 핵심 요인.

KV-Cache 폭발, 컨텍스트 길이, 그리고 Paged Attention
KV-Cache 메모리 계산식
KV_bytes = 2 × n_layers × n_heads × d_head × seq_len × batch × dtype_bytes

// Llama 3 70B (FP16), 128K context, batch=1
= 2 × 80 × 8 × 128 × 131,072 × 1 × 2
≈ 42.9 GB (KV-Cache만)
⚠
H100 80GB에서 Llama 3 70B: 가중치 FP16 = 140GB → 불가. INT4 양자화 시 35GB, KV-cache 포함 시 80GB 한계 도달 (context 제한)
✓
H200 141GB에서: INT4 양자화 Llama 3 70B (35GB) + KV-cache 42.9GB (128K) = 77.9GB → 128K 컨텍스트 처리 가능
📄
Paged Attention (vLLM): KV-cache를 고정 크기 페이지(Block) 단위로 가상 메모리처럼 관리. 단편화 최소화, 여러 시퀀스 간 KV 공유(prefix caching) 가능 → 실효 처리량 2~4× 향상

KV-Cache는 Transformer의 Key/Value 행렬을 미리 계산해 재사용. 없으면 매 토큰마다 전체 컨텍스트 재계산. 컨텍스트가 길수록 KV-Cache가 커짐 — 메모리 용량이 직접적으로 컨텍스트 길이 한계를 결정.

n_heads × d_head = hidden_dim (예: 70B에서 8192). Grouped Query Attention(GQA, Llama 3)에서 KV heads(8개) < Q heads(64개) → KV-Cache 크기 8× 절약. MQA(Multi-Query Attention)는 더 공격적으로 KV heads=1. 컨텍스트 2× → KV-Cache 2× → 메모리 직선 비례.

기존 방식: 시퀀스 최대 길이만큼 연속 메모리 예약 → 실제 길이가 짧으면 낭비(단편화). Paged Attention: 16토큰 단위 블록으로 동적 할당. 블록 테이블로 비연속 물리 메모리 매핑. Prefix Sharing: 같은 시스템 프롬프트 가진 요청들이 KV 블록 공유 → GPU 메모리 사용량 최대 50% 절감.

CXL 3.0 메모리 풀링

PCIe 6.0 기반 Coherent Interconnect. CPU, GPU, 가속기가 동일 메모리 풀에 캐시 코히런트 접근.

128 TB 메모리 풀을 여러 노드가 공유 — 대형 LLM 가중치를 단일 노드에 국한하지 않음.

Fabric Manager로 메모리 동적 할당. 사용률 낮은 서버의 DRAM을 타 서버가 활용.
