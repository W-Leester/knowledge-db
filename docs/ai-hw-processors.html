SLIDE 01 · AI 워크로드

왜 CPU 하나로는 부족한가

AI 연산의 본질
  • AI = 거대한 행렬 곱셈의 반복 (GEMM)
  • GPT-3 학습: 3.14 × 10²³ FLOP 필요
  • A100 GPU 312 TFLOPS → 약 11.6일
  • 단일 CPU로 하면 → 수십 년 이상
  • 병렬 연산 능력 = AI 인프라의 핵심 지표
순차 처리 (CPU): 작업을 하나씩 처리. 복잡한 로직에 강함.
병렬 처리 (GPU): 수천 개 코어가 동시에 단순 연산. 행렬곱에 최적.
Amdahl's Law: 병렬화 불가 부분이 병목. 행렬곱은 99%+ 병렬화 가능.
Roofline Model: 연산 강도(FLOPs/byte) < 하드웨어 능선 → Memory-bound

SLIDE 03 · GPU

GPU — 병렬의 제왕

SIMT — Single Instruction Multiple Threads
  • 수천 개 스레드가 동일 명령어를 동시 실행
  • 하나의 SM(Streaming Multiprocessor)에 128개 CUDA 코어
  • H100: 132 SM × 128 코어 = 16,896 CUDA 코어
  • 행렬 A×B의 각 원소를 독립적으로 병렬 계산
  • Shared Memory로 SM 내 데이터 재사용
왜 GPU가 AI에 최적인가
  • GEMM: 완전한 데이터 병렬성 (data-parallelism)
  • HBM3 메모리 → 3.35 TB/s 대역폭
  • NVLink 4.0: GPU 간 900 GB/s 직결 통신
  • CUDA 생태계: cuBLAS, cuDNN, TensorRT
Warp & 레이턴시 숨기기 (Latency Hiding)
  • Warp = 32개 스레드의 실행 단위
  • 한 Warp가 메모리 대기 시 → 즉시 다른 Warp 실행
  • 스레드 Context Switch 비용 = 0 (하드웨어 스케줄러)
  • 충분한 Occupancy가 레이턴시 숨기기의 핵심
  • Warp Divergence: if/else가 성능을 절반으로
Tensor Core & FP8 혁명
  • Volta(V100): FP16 GEMM 전용 코어 등장
  • Ampere(A100): TF32, BF16, INT8 지원
  • Hopper(H100): FP8 도입 → 3,958 TFLOPS
  • FP8 = FP16 대비 2× 처리량, 정밀도 손실 최소
  • 4차원 GEMM: D = A×B + C (행렬 누산 연산)
Flash Attention — IO 최소화 알고리즘
  • 표준 Attention: Q·Kᵀ를 HBM에 저장 → O(N²) IO
  • Flash Attention: SRAM(192KB/SM) 내에서 타일 연산
  • HBM 접근 횟수 5~10× 감소 → 속도 3× 향상
  • LLM 추론은 Memory-bound → 대역폭이 핵심
  • V3: 128K context에서도 선형 메모리 사용

SLIDE 04 · NPU

NPU — 추론 효율의 극한

Apple M4 Neural Engine: 38 TOPS, 소비전력 ~3W.
TOPS/W 기준으로 H100의 10배 이상 효율.
Unified Memory 아키텍처
CPU/GPU/NPU가 동일 메모리 공유. 데이터 복사 없음. Apple Silicon의 핵심 혁신.
Systolic Array — 메모리 없는 연산
  • MAC(Multiply-Accumulate) 유닛이 격자 형태로 배열
  • 데이터가 물결처럼 흘러 연산 — 추가 메모리 접근 없음
  • 가중치(Weight)를 배열에 고정 → Stationary 방식
  • 높은 데이터 재사용률 → 에너지 효율 극대화
  • Google TPU, Apple Neural Engine 모두 사용
양자화 (Quantization) 효과
  • FP32 → INT8: 메모리 4× 절감, 연산 4× 빠름
  • FP32 → INT4: 메모리 8× 절감 (AWQ, GPTQ 기법)
  • NPU는 INT8/INT4 전용 회로 → GPU 대비 더 효율적
  • QAT(Quantization-Aware Training)로 정확도 유지
  • 온디바이스 LLM: Llama 3.2 3B INT4 = 1.8GB

SLIDE 05 · TPU

TPU — Google이 직접 만든 이유

TPU 세대별 진화
  • v1 (2015): 추론 전용, 92 TOPS INT8
  • v2 (2017): 학습 추가, HBM 도입
  • v4 (2021): 275 TFLOPS BF16
  • v5p (2023): 459 TFLOPS, 4,096칩 Pod
  • v5e (2023): 비용 최적화 버전
XLA 컴파일러 의존 → 범용성 낮음
BUT 특화 성능 극대화 & Google 인프라와 최적 통합
TPU Systolic Array 원리
  • 128×128 MAC 유닛 배열 (v1 기준)
  • 데이터가 행/열 방향으로 펄스처럼 이동
  • 각 MAC은 이전 결과를 누산하며 통과
  • 메모리 접근 없이 칩 내에서 연산 완료
  • Weight Stationary: 가중치 고정, 입력 이동
  • BF16: FP32 대비 4× 처리량, 범위 유지
ICI vs NVLink — 상호연결 비교
  • ICI (Inter-Chip Interconnect): TPU 전용, 3D 토러스 토폴로지
  • TPU Pod: 4,096칩이 직접 연결 (no switch)
  • ICI 대역폭: 192 GB/s per link (v4)
  • NVLink 4.0: NVIDIA GPU 간, 900 GB/s 양방향
  • NVSwitch 기반 full-mesh → 더 유연한 토폴로지
  • TPU ICI: 낮은 레이턴시, 규모 확장성 우수

Section 00

AI를 움직이는 두뇌

프로세서 아키텍처 완전 분해 — CPU · GPU · NPU · TPU에 대한 심층 기술 분석 자료입니다. 전문가를 대상으로 기초 개념부터 최신 아키텍처 혁신까지 다룹니다.

Section 01

왜 CPU 하나로는 부족한가

AI 연산의 핵심은 거대한 행렬 곱셈(GEMM)의 반복입니다. GPT-3 학습에는 3.14 × 10²³ FLOP이 필요하며, A100 GPU(312 TFLOPS)로도 약 11.6일이 걸립니다. 단일 CPU로는 수십 년이 소요됩니다.

Amdahl's Law에 따르면 병렬화 가능 비율이 높을수록 병렬 처리의 이점이 커집니다. 행렬 곱셈은 99%+ 병렬화가 가능하므로 GPU의 수천 개 코어가 극적인 성능 향상을 제공합니다.

Roofline Model: 연산 강도(FLOPs/byte)가 하드웨어 능선보다 낮으면 Memory-bound → LLM 추론이 여기에 해당
Section 02

CPU — 범용의 강자, AI의 조력자

CPU는 복잡한 제어 흐름, Branch Prediction, Out-of-Order Execution이 강점입니다. Intel Sapphire Rapids의 AMX(Advanced Matrix Extensions)는 2D Tile 레지스터로 최대 4 TOPS INT8 행렬 연산을 지원합니다.

LLM 추론에서 CPU의 역할: KV-cache 관리, sparse 연산, 배치 전처리. GPU/NPU/TPU의 조력자.
Section 03

GPU — 병렬의 제왕

SIMT(Single Instruction Multiple Threads) 아키텍처로 수천 개 스레드가 동일 명령어를 동시 실행합니다. NVIDIA H100은 132 SM × 128 CUDA 코어 = 16,896 CUDA 코어를 탑재합니다.

Hopper 아키텍처의 FP8 Tensor Core는 3,958 TFLOPS를 제공하며, HBM3 메모리가 3.35 TB/s 대역폭을 지원합니다. Flash Attention V3는 HBM 접근을 최소화해 레이턴시를 3× 줄입니다.

LLM 추론은 Compute-bound가 아닌 Memory-bound → 메모리 대역폭이 성능의 핵심
Section 04

NPU — 추론 효율의 극한

NPU는 INT8/INT4 전용 하드웨어로 엣지 추론에 특화됩니다. Apple M4 Neural Engine은 38 TOPS를 약 3W로 처리하며, TOPS/W 기준으로 H100보다 10배 이상 효율적입니다. Unified Memory 아키텍처는 CPU-GPU-NPU 간 데이터 복사 오버헤드를 제거합니다.

FP32 → INT4 양자화: 메모리 8× 절감. Llama 3.2 3B INT4 모델 = 1.8GB → M4 MacBook에서 실행 가능
Section 05

TPU — Google이 직접 만든 이유

2015년 추론 전용 TPU v1(92 TOPS) 이후, TPU v5p는 459 TFLOPS BF16으로 성장했습니다. 4,096칩이 3D 토러스 ICI로 직결되는 Pod 구조가 초대형 모델 학습을 가능하게 합니다.

트레이드오프: XLA 컴파일러 의존 → 범용성 낮음 vs PyTorch 대비 특화 워크로드에서 2-3× 성능 우위
