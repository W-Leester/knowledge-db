# KnowledgeDB App — 기술 방식 비교

콘텐츠가 "이미 완성된 HTML"이므로, 앱의 본질은 **HTML을 잘 보여주고, 새 문서를 쉽게 배달하는 것**이다.
후보는 크게 4가지.

## 옵션 비교

| | A. PWA (웹앱) | B. Capacitor 래핑 앱 | C. Flutter + WebView | D. 네이티브 (Kotlin) |
|---|---|---|---|---|
| 형태 | GitHub Pages 등에 호스팅, 홈 화면에 추가하면 앱처럼 동작 | 웹(A와 동일 콘텐츠)을 껍데기 앱으로 감싼 진짜 앱 | Flutter 앱 안에서 WebView로 HTML 표시 | Android 앱 + WebView |

## 추천

**1순위: A. PWA로 시작.**

근거:
- 문서가 이미 웹 기술(HTML)이므로 변환 비용이 0에 가깝다.
- GitHub 저장소에 push → 몇 초 뒤 폰에서 새 문서가 보이는 **가장 단순한 업데이트 파이프라인** (NFR-04, NFR-05 충족).
- 개인용 1인 사용이라 스토어 배포·네이티브 기능(알림, 센서 등)이 필요 없다.
- Android/iOS 어느 쪽이든 동작한다 (D1 결정을 기다릴 필요가 적음).
- 나중에 부족하면 **같은 콘텐츠를 그대로 Capacitor(B)로 감싸 승격** 가능 — 지금 결정이 미래를 막지 않는다.

**B(Capacitor)를 처음부터 택할 이유가 있는 경우:**
- 홈 화면 아이콘/스플래시/전체화면 등 "앱다운 외형"이 중요하거나,
- 인터넷 없이 완전 오프라인 우선으로 쓰고 싶거나 (파일을 APK에 내장),
- 나중에 스토어 등록 생각이 있는 경우.

**전제 조건 (A 선택 시):**
- GitHub 계정 + public 저장소 (private도 가능하나 Pages는 유료 플랜 필요)
- 폰트 로컬 번들 및 `ai-hw-processors.html` 모바일 반응형 보완 (Phase 1에서 처리)

> 민감한 내용이 담긴 문서라면 public 호스팅이 부적절할 수 있음 → 체크리스트 D3 참고.
